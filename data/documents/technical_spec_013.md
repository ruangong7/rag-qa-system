# 缓存规范 

## 规范范围
本技术规范适用于 `edge-api` 和 `platform-shared-lib` 系统中的缓存机制。该规范定义了使用 Redis 7 的具体要求，包括必须实现的机制、参数以及验证信号。本规范不适用于人力资源审批流程（HR approval）和法律保留政策（legal retention）。

## MUST 要求
- **MUST**：所有缓存实现必须使用 Redis 7。
- **MUST**：Redis 7 的内存淘汰策略必须设置为 `allkeys-lru`。
- **MUST**：所有缓存项必须设置 TTL（Time To Live），以确保数据不会无限期驻留在缓存中。

## 配置项
- **Redis 版本**：Redis 7
- **内存淘汰策略**：`allkeys-lru`
- **TTL 设置**：根据业务需求设置合理的 TTL 值，确保数据在合理的时间内过期。

## 接口行为
- **GET 请求**：从缓存中获取数据时，应检查数据是否存在且未过期。如果数据不存在或已过期，则应从后端数据库中获取数据并更新缓存。
- **SET 请求**：向缓存中写入数据时，必须设置 TTL。例如：
  ```bash
  SET key value EX ttl
  ```
- **DEL 请求**：删除缓存中的数据时，应确保数据被正确移除，并触发相应的清理机制。

## 测试要求
- **MUST**：工程师在发布前必须测试 Redis 7 的缓存机制，确保其符合规范要求。
- **MUST**：测试过程中必须验证 `allkeys-lru` 内存淘汰策略的有效性。
- **MUST**：测试过程中必须验证所有缓存项是否设置了合理的 TTL。

## 失败处理
- **缓存失效**：如果缓存中的数据因 TTL 过期而失效，系统应自动从后端数据库中重新加载数据并更新缓存。
- **内存不足**：当 Redis 由于内存不足而触发 `allkeys-lru` 淘汰策略时，系统应记录日志并监控被淘汰的数据量，以便进行后续优化。

## 禁止事项
- **禁止**：禁止在生产环境中使用未经测试的 Redis 版本。
- **禁止**：禁止在缓存中存储敏感信息，如用户密码等。
- **禁止**：禁止在缓存配置中禁用 TTL，以防止数据无限期驻留。

通过以上规范，确保 `edge-api` 和 `platform-shared-lib` 系统中的缓存机制能够高效、安全地运行。
