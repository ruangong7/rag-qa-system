# 认证流程

## 架构上下文
在我们的企业平台中，认证流程是一个关键的安全组件，主要用于确保用户身份的验证和授权。本次架构评审主要关注OIDC登录（OIDC login）的设计，并由Search Platform团队负责审查运行时边界和依赖关系。

## 组件边界
认证流程的核心组件包括：
- **OIDC Provider**：负责处理OpenID Connect协议的认证请求。
- **JWT Token Service**：生成和验证JSON Web Tokens (JWT)。
- **Refresh Token Service**：管理刷新令牌（refresh token），用于获取新的访问令牌。

这些组件共同协作，确保用户能够安全地登录并获得所需的访问权限。

## 运行时调用流
1. **用户发起登录请求**：用户通过前端应用或API网关发起登录请求。
2. **重定向到OIDC Provider**：请求被重定向到OIDC Provider进行身份验证。
3. **身份验证**：用户在OIDC Provider处完成身份验证。
4. **生成JWT Access Token**：成功验证后，OIDC Provider生成JWT访问令牌（JWT access token）。
5. **返回令牌**：JWT访问令牌和刷新令牌（refresh token）被返回给用户。
6. **使用访问令牌**：用户使用JWT访问令牌访问受保护的资源。

## 依赖
认证流程依赖于以下系统和服务：
- **Kubernetes**：用于部署和管理认证相关的微服务。
- **PostgreSQL**：存储用户信息和令牌数据。
- **API Gateway**：作为前端应用和认证服务之间的中介。

## 中断/降级
在认证流程中，如果某个组件出现故障，系统将采取以下措施：
- **OIDC Provider不可用**：系统将暂时禁用新用户的登录请求，并引导用户稍后再试。
- **JWT Token Service不可用**：系统将使用缓存中的有效令牌继续提供服务，直到服务恢复。
- **Refresh Token Service不可用**：系统将延长现有访问令牌的有效期，以减少对刷新令牌的依赖。

## 反模式
在设计认证流程时，应避免以下反模式：
- **硬编码凭证**：不要在代码中硬编码任何敏感信息，如密钥或令牌。
- **单一因素认证**：仅使用密码进行认证是不安全的，应采用多因素认证（MFA）。
- **无限期有效的令牌**：访问令牌应设置合理的过期时间，以减少安全风险。

## 不适用范围
本文档不适用于以下场景：
- 员工手册规则
- 隐私请求SLA

通过以上架构设计，我们确保了认证流程的安全性和可靠性，同时明确了各组件的职责和依赖关系。
