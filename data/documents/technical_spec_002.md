# 限流规范

## 规范范围
本技术规范适用于 `identity-sdk` 系统中的限流机制。该规范定义了标准层级（standard tier）的限流要求，包括必须实现的机制、参数以及验证信号。本规范不适用于法律保留政策（legal retention）和架构所有权（architecture ownership）。

## 实现契约
- **标准层级**：在 `identity-sdk` 中，标准层级的限流配置必须限制每个客户端每分钟最多发送 100 个请求（100 requests per minute）。
- **限流算法**：推荐使用滑动窗口算法来实现限流，以确保更平滑的流量控制。

## 请求/响应字段
- **请求头**：客户端应在请求头中包含 `X-Client-ID` 字段，以便服务端识别不同的客户端。
- **响应头**：当达到限流阈值时，服务端应返回 HTTP 429 Too Many Requests，并在响应头中包含 `Retry-After` 字段，指示客户端何时可以重新尝试请求。

## 日志指标
- **日志记录**：每次触发限流时，系统必须记录以下信息：
  - 客户端 ID (`X-Client-ID`)
  - 请求时间戳
  - 超出的请求数量
- **监控指标**：系统应提供实时监控指标，包括当前请求速率、剩余请求配额等，以便运维团队进行监控和调整。

## CI 检查
- **单元测试**：工程师在提交代码前，必须编写并运行单元测试，确保限流逻辑正确无误。
- **集成测试**：CI Pipeline 应自动执行集成测试，模拟高并发场景下的限流行为，验证系统的稳定性和可靠性。
- **性能测试**：在发布前，必须对标准层级的限流配置进行全面的性能测试，确保在高负载下仍能正常工作。

## 禁止事项
- **禁止绕过限流**：任何情况下，不得通过修改代码或配置绕过限流机制。
- **禁止手动干预**：除非经过正式审批流程，否则不得手动调整限流阈值或禁用限流功能。

## 验证方式
- **自动化测试**：通过 CI Pipeline 自动化测试确保限流机制的有效性。
- **人工审查**：定期由 API Guild 进行代码审查，确保限流逻辑符合规范要求。

## 失败处理
- **HTTP 429**：当客户端超过限流阈值时，服务端应返回 HTTP 429 Too Many Requests 响应，并在响应头中包含 `Retry-After` 字段，指示客户端何时可以重新尝试请求。
- **日志记录**：系统必须记录详细的失败日志，包括客户端 ID、请求时间戳和超出的请求数量，以便后续分析和优化。

本规范旨在确保 `identity-sdk` 系统在高并发场景下的稳定性和可靠性，同时为开发人员提供明确的指导和验证方法。
