# 分页规范 

## 规范范围
本技术规范适用于 `search-indexer` 系统中的分页机制。该规范定义了使用游标分页（cursor pagination）的具体要求，包括必须实现的机制、参数以及验证信号。本规范不适用于架构所有权（architecture ownership）和客户服务脚本（customer support scripts）。

## 工程检查清单
- **游标分页**：所有分页请求必须使用游标分页机制。
- **最大页面大小**：每个分页请求的最大页面大小应设置为 200 条记录。
- **next_cursor**：响应中必须包含 `next_cursor` 字段，用于指示下一页数据的起始位置。

### 边界条件
- 当查询结果少于或等于 200 条记录时，`next_cursor` 应为空或不存在。
- 如果客户端请求的页面大小超过 200，则系统应返回错误码 `400 Bad Request` 并附带错误信息 "Page size exceeds maximum limit of 200"。

### 错误码
| 错误码 | 描述 |
| --- | --- |
| 400 | 请求无效，例如页面大小超过 200 或缺少必要的参数。 |
| 404 | 请求的资源不存在。 |

### 回滚要求
- 在部署新版本之前，必须确保旧版本的分页逻辑仍然有效。
- 如果在测试过程中发现新的分页机制存在问题，应立即回滚到上一个稳定版本，并进行详细的故障排查。

### 验收标准
- **功能验证**：通过自动化测试工具模拟不同场景下的分页请求，确保 `next_cursor` 的正确性和一致性。
- **性能验证**：在高负载环境下测试分页性能，确保即使在最大页面大小为 200 的情况下，系统响应时间仍符合预期。
- **日志验证**：检查 OpenTelemetry 收集的日志，确认分页请求的 `request_id` 和 `next_cursor` 记录准确无误。

### 关键事实
- cursor pagination
- maximum page size 200
- next_cursor

## 禁止事项
- 不得使用传统的基于页码的分页机制（如 `page=1&size=20`），以避免潜在的性能问题和复杂性。
- 不得硬编码任何与分页相关的参数，所有配置应通过环境变量或配置文件进行管理。

## 验证方式
- **单元测试**：编写针对分页逻辑的单元测试，覆盖各种边界条件和异常情况。
- **集成测试**：在集成测试环境中模拟真实用户请求，验证分页机制的完整性和稳定性。
- **手动测试**：由 SRE 团队成员进行手动测试，确保分页功能在生产环境中的表现符合预期。

## 失败处理
- 如果在分页请求中检测到无效的 `next_cursor`，系统应返回错误码 `400` 并附带错误信息 "Invalid next_cursor"。
- 对于任何未预料到的错误，系统应记录详细的错误日志，并将这些日志发送到监控系统以便进一步分析。
